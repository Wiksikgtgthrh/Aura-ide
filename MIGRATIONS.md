# 📦 Руководство по миграциям БД

## 🚀 Быстрый старт

### Для нового пользователя (первая установка)

**Самый простой способ** — запустить единую команду, которая всё проверит и инициализирует:

```bash
pnpm setup
```

Эта команда:
- ✅ Проверит наличие `.env.development.local`
- ✅ Проверит обязательные переменные окружения (`DATABASE_URL`, `OPENAI_API_KEY`, `BETTER_AUTH_SECRET`)
- ✅ Автоматически применит миграции БД
- ✅ Покажет полезные сообщения об ошибках

Если что-то не хватает, скрипт подскажет, что нужно добавить.

---

### Для опытных: ручная инициализация

#### 1. Установка переменной окружения
Убедитесь, что `DATABASE_URL` установлена в `.env.development.local`:

```bash
# PostgreSQL локально
DATABASE_URL="postgresql://user:password@localhost:5432/mydb"

# Или для удаленной БД (например, Neon)
DATABASE_URL="postgresql://user:password@db.neon.tech:5432/mydb"
```

#### 2. Инициализация БД
Создает все таблицы из начальной миграции:

```bash
pnpm db:init
```

Это выполнит:
- Создание таблиц для auth (user, session, account, verification)
- Создание таблиц приложения (projects, apiKeys, chats, messages, preferences)
- Установка foreign key связей

### 3. Применение миграций
Если вы добавили новые миграции в папку `drizzle/`:

```bash
pnpm db:migrate
```

## 📝 Создание новой миграции

### Шаг 1: Обновите схему
Отредактируйте `/lib/db/schema.ts` - добавьте новые таблицы или измените существующие:

```typescript
export const newTable = pgTable('new_table', {
  id: serial('id').primaryKey(),
  name: text('name').notNull(),
  createdAt: timestamp('createdAt').notNull().defaultNow(),
})
```

### Шаг 2: Сгенерируйте SQL миграцию
```bash
pnpm db:generate
```

Это создаст новый файл в папке `drizzle/` с SQL командами.

### Шаг 3: Проверьте сгенерированный SQL
Откройте новый файл в `drizzle/` и убедитесь, что миграция корректна.

### Шаг 4: Примените миграцию
```bash
pnpm db:push
# или
pnpm db:migrate
```

## 📊 Структура миграций

```
drizzle/
├── meta/
│   ├── _journal.json          # Журнал миграций
│   └── 0000_snapshot.json     # Снимок схемы
├── 0000_regular_clint_barton.sql  # Начальная миграция
└── 0001_your_migration.sql        # Новая миграция
```

## 🔧 Доступные команды

| Команда | Описание |
|---------|---------|
| `pnpm db:init` | Инициализирует БД с начальной миграцией |
| `pnpm db:generate` | Генерирует новую миграцию на основе изменений schema.ts |
| `pnpm db:push` | Применяет миграции напрямую к БД (Drizzle Kit) |
| `pnpm db:migrate` | Применяет миграции из папки drizzle/ |

## ⚠️ Важные моменты

### Использование переменных окружения
- **Локальная разработка**: `.env.local`
- **Production**: установите `DATABASE_URL` в переменных среды Vercel
- **Тестирование**: используйте отдельную БД

### Безопасность
- Никогда не коммитьте реальные `DATABASE_URL` с паролями
- Используйте переменные окружения для хранения credentials
- Убедитесь, что права доступа к БД минимально необходимы

### Откат миграции
Drizzle Kit не поддерживает автоматический откат. Если нужно откатить:

1. Создайте новую миграцию с обратными изменениями
2. Или восстановите БД из резервной копии

## 🆘 Решение проблем

### "DATABASE_URL не установлена"
```bash
# Создайте .env.local
echo 'DATABASE_URL="postgresql://..."' > .env.local
```

### "Таблица уже существует"
Это нормально при повторном запуске `db:init`. Скрипт пропускает существующие таблицы.

### Ошибка подключения
1. Проверьте, что PostgreSQL сервер запущен
2. Убедитесь в корректности `DATABASE_URL`
3. Проверьте права доступа пользователя БД

### Несинхронизированная схема
Если `schema.ts` не соответствует действительной схеме БД:

```bash
# Обновите schema.ts вручную или используйте
pnpm db:generate  # для создания миграции изменений
```

## 🔑 Управление переменными окружения

### Приоритет загрузки .env файлов

Скрипты БД (`db:init`, `db:migrate`, `setup`) автоматически читают переменные в этом порядке:

1. **`.env.development.local`** — локальные настройки разработчика (приоритет 1, не коммитить в git)
2. **`.env.local`** — локальные переменные (не коммитить в git)
3. **`.env`** — общие переменные (можно коммитить в git)

Первый найденный файл с переменной используется, остальные пропускаются.

### Как получить BETTER_AUTH_SECRET

```bash
openssl rand -base64 32
```

Вставьте результат в `.env.development.local`:
```
BETTER_AUTH_SECRET="генерированная_строка_здесь"
```
