# ⚡ Ускорение навигации, фикс авторизации и новый IDE

Список изменений этой итерации. Всё проверено: `next build` проходит (22/22
страниц, Partial Prerender), e2e-прогон auth + чата на локальном PostgreSQL.

## 🚀 Производительность

### Вход / регистрация
- **`app/sign-in/page.tsx`, `app/sign-up/page.tsx`** — проверка сессии (поход в БД)
  перенесена внутрь `<Suspense>`: форма рисуется мгновенно из статичного шелла,
  редирект авторизованных стримится фоном. TTFB ~120 мс вместо блокировки на БД.
- **`components/auth-form.tsx`** — после успешного входа навигация происходит сразу
  (без искусственных 600 мс), убран лишний `router.refresh()` (двойная работа
  сервера); на маунте формы префетчится `/`, чтобы главная была готова к моменту
  входа.

### Гостевой вход «со второй попытки»
- Причина: старый код делал `signOut()` → `signIn.anonymous()`. При «залипшей»
  anonymous-сессии better-auth отвечает
  `ANONYMOUS_USERS_CANNOT_SIGN_IN_AGAIN_ANONYMOUSLY` — первый клик падал, второй
  срабатывал. Воспроизведено e2e.
- Фикс: прямая попытка `signIn.anonymous()`, при ошибке — автоматический
  `signOut()` + повтор в том же клике.
- **`lib/auth.ts`** — cookies `SameSite=None; Secure` теперь ставятся только в
  v0-runtime (iframe-превью). На своём сервере/локально — `SameSite=Lax`: раньше
  по plain-http браузер отбрасывал Secure-куки и сессия «не ставилась».

### Переходы по разделам
- **`lib/navigation-context.tsx`** — разделы, смонтированные в `<Activity>`-шеллах
  (`/`, `/projects`, `/teams`, `/plugins`, `/settings`), переключаются через
  `history.pushState` — ноль сетевых запросов вместо серверного RSC-раундтрипа
  на каждый клик. Действует и из чата (внутри app-шелла).
- **`components/home-content.tsx`** — переход «главная → чат» не ждёт server
  action `createChat` (страница чата и так ретраит гонку) — навигация мгновенная.
- **`components/chat-view.tsx`** — иконки «Плагины/Навыки» слева от чата больше не
  `<a href>` (полная перезагрузка!), а мгновенное переключение секции настроек.
- Секции внутри настроек уже были клиентским состоянием — подтверждено, мгновенно.

### cacheComponents / сборка
- **`app/(app)/layout.tsx`** — `getSession()` и `SettingsProvider` (использует
  `usePathname()`) перенесены внутрь `<Suspense>`; иначе `next build` падал на
  пререндере динамических маршрутов.
- **Все динамические страницы** (`/chat/[id]`, `/teams/[teamId]`,
  `/plugins/[slug]`, `/my-api`, `/free`, `/teams/invite/[token]`) — данные
  загружаются в Suspense-ветке: статичный скелет приходит мгновенно, контент
  стримится. Сборка: все маршруты ◐ Partial Prerender.

## 🤖 Чат и генерация проекта

- **`app/api/chat/route.ts`**:
  - Дизайн-интервью: для нового проекта модель обязана сначала спросить стиль
    дизайна и выдать блок `<design-choices>Вариант|…|Другой</design-choices>`,
    и только после ответа сгенерировать полную файловую иерархию. Состояние
    проекта (`PROJECT STATE: NEW/EXISTING`) вычисляется на сервере по наличию
    ```file:-блоков — надёжно, не зависит от «памяти» модели.
  - **Баг-фикс**: пользовательские API-ключи с кастомным `baseUrl` (Groq,
    OpenRouter, LM Studio и т.д.) не работали — провайдер по умолчанию ходит в
    OpenAI Responses API. Заменено на `openai.chat(...)` (Chat Completions).
- **`components/chat-view.tsx`** — блок `<design-choices>` рендерится кликабельными
  чипами; сырой тег скрыт из текста сообщения.

## 🖥️ IDE-панель (`components/ide-panel.tsx`)

- **Превью переписано** (`lib/preview-runtime.ts`): вместо regex-«транспилятора»
  (ломался на дженериках, интерфейсах, union-типах — причина «кривого» превью) —
  настоящий Babel Standalone в iframe (пиновая версия 7.26.4 с CDN),
  CommonJS-реестр модулей, поддержка `@/`-алиасов, `.css`-файлов, fallback для
  неизвестных lucide-иконок.
- Файлы доставляются в iframe через `postMessage` — превью обновляется живьём
  без перезагрузки iframe при каждом нажатии клавиши.
- **Оверлей ошибок**: компиляция, рантайм, unhandled rejection — красная панель
  с трейсом + дублирование в консоль.
- **Консоль**: перехват `console.*` из превью, счётчик ошибок на кнопке, очистка,
  REPL-ввод (выполнение JS прямо в контексте превью).
- **Файловое дерево**: создание файлов и папок (в т.ч. вложенных), переименование,
  удаление (папки — с подтверждением), пустые папки поддерживаются. Локальные
  правки и удаления переживают стриминг от модели.
- Monaco: язык подсветки по расширению файла (ts/tsx/js/css/json/html/md).

## 📦 Publish (`components/publish-dialog.tsx`)

Проект упаковывается в полноценное Vite + React + TS + Tailwind приложение
(`lib/project-scaffold.ts`), для HTML-режима — статический сайт под nginx.

- **GitHub** — публикация через API (PAT со scope `repo`, хранится только в
  браузере): создание репозитория + один коммит со всеми файлами
  (`app/actions/publish.ts`). Повторная публикация — новый коммит.
- **Свой сервер (SSH)** — браузер не может открыть SSH-соединение, поэтому
  скачивается deploy-бандл: проект + Dockerfile + docker-compose + `deploy.sh`,
  преднастроенный на ваши host/user/port/path (rsync + `docker compose up -d
  --build` одной командой).
- **Aura Hosting (Docker + VPS)** — Docker-бандл на проект: multi-stage
  Dockerfile (node build → nginx), `docker-compose.yml`, `nginx.conf` (gzip,
  SPA-fallback, кеш статики). Требования: VPS, Docker 24+, compose v2.
- ZIP собирается на клиенте без зависимостей (`lib/zip.ts`), `deploy.sh` получает
  исполняемый бит.

## 🗄️ База данных

- **`lib/db/index.ts`** — drizzle-драйвер выбирается автоматически: Neon → neon-http,
  любой другой PostgreSQL (VPS/docker) → node-postgres. Раньше проект запускался
  только с Neon.
- Быстрые фиксы типов: `revalidateTag(x, 'max')` в billing/mcp/memories,

## Критичные баги

- **«Новые чаты нельзя начать при активной сессии»** (`components/home-content.tsx`):
  главная живёт в постоянном Activity-шелле, флаг `creating` выставлялся один

  принудительная генерация проекта. Проверено e2e (три хода, без цикла).
- **Скрытая потеря истории между ходами** (`lib/chat-store.ts`): `loadChatMessages`
  кэшировался через `unstable_cache`, а `revalidateTag` из фонового `onEnd`

  (включая голые ```tsx и незакрытые при стриминге), промпт ужесточён — код
  только внутри ```file: блоков.

# 🚀 Итерация 3 — обязательная пятёрка

## 1. Персистентная файловая система проекта
- Новая таблица `project_files` (chatId+path PK, каскад при удалении чата) —
  миграция `drizzle/0004_project_files.sql` (идемпотентна; для существующей БД
  достаточно `pnpm db:push` или `node scripts/init-db.mjs`).
- Правки в Monaco автосохраняются (debounce 1.2 с, полная синхронизация вместе
  с удалениями) с индикатором «Сохранение… / Сохранено» в тулбаре. F5 больше
  не теряет работу.
- Ответы модели апсертятся в БД прямо из `onEnd` стрима — даже если вкладку
  закрыли во время генерации.
- При открытии чата редактор сидится из БД; старые file-блоки из истории
  сообщений больше не перезатирают сохранённые правки (baseline-дифф).
- **Модель видит ручные правки**: в контекст каждого хода EXISTING-проекта
  подставляется секция CURRENT PROJECT FILES (лимит 48КБ) — проверено e2e
  маркером в контенте.

## 2. Лимиты и учёт токенов
- `lib/limits.ts` + запись `token_usage` из `onFinish` стрима (prompt/completion
  токены, modelId, apiKeyId). Экран «Использование» теперь получает реальные данные.
- Дневные лимиты ТОЛЬКО для встроенных Aura-моделей (Gateway, платит владелец):
  гость — 20/день, аккаунт — 100/день. **Собственные ключи не ограничиваются**:
  ключи строго по-аккаунтные (у гостей тоже свои), их использование пишется с
  `apiKeyId` и в лимит не входит. Превышение → 429 с подсказкой добавить свой
  ключ; в чате — понятное сообщение со ссылкой на «Мои API».
- Проверено e2e: 20/20 → 429; тот же гость со своим ключом → 200 без лимита.

## 3. Восстановление пароля
- Страницы `/forgot-password` и `/reset-password` (Suspense для useSearchParams),
  ссылка «Забыли пароль?» в форме входа. Полный цикл проверен e2e: запрос →
  письмо (fallback-лог) → смена пароля по токену → вход с новым паролем.

## 4. Retry / Regenerate / автоназвание
- Кнопка «Повторить» в блоке ошибки и «Перегенерировать» под последним ответом
  (useChat.regenerate).
- Серверный дедуп при regenerate: если id сообщения уже есть в истории,
  сохранённая копия и всё после неё отбрасываются — модель не видит дублей.
- Автоназвание чата после первого обмена: короткий заголовок генерируется той
  же моделью (best-effort, при ошибке остаётся старое название). Проверено e2e.

## 5. Мобильный IDE
- Плавающий переключатель «Чат / Превью» на телефонах — раньше панель превью
  была недостижима (тоггл жил внутри скрытой панели).

# 🔒 Итерация 6 — security hardening + оптимизации

## Безопасность
- **SSRF-гард** (lib/ssrf.ts): пользовательский baseUrl валидируется до
  запроса — только http/https, без кред в URL, DNS резолвится и все адреса
  должны быть публичными. Блок loopback, 10/172.16-31/192.168, 169.254
  (метаданные), CGNAT 100.64/10, ::1, fe80, fc00/fd00, IPv4-mapped. Применён
  в verifyKey/probeModel и в /api/chat для обоих резолверов ключа (иначе ответ
  внутреннего сервиса утекал в стрим). 4 юнит-теста.
- **Заголовки безопасности + CSP** (next.config.mjs): CSP с frame-ancestors
  none, object-src none, base-uri/form-action self (+ allowlist CDN и
  unsafe-inline/eval для inline-скрипта темы и Babel в превью), плюс
  X-Frame-Options DENY, nosniff, Referrer-Policy, HSTS, Permissions-Policy.
  Проверено curl — все заголовки отдаются.
- **Рейт-лимит** (lib/auth.ts, better-auth): 60/мин/IP по умолчанию + правила:
  sign-up 10/час, sign-in 10/мин, гость 15/час, сброс пароля 5/час.
- **Перечисление пользователей** (app/api/teams/search): гостям запрещено,
  минимум 3 символа, троттлинг 20 запросов/10с на пользователя (429).
  Проверено e2e: гость → пусто, <3 → пусто, 21-й → 429, по чужому логину — ок.
