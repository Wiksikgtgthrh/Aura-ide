# 🏗️ Архитектура системы инициализации

## Диаграмма потока инициализации

```
┌─────────────────────────────────────────────────────────────┐
│                    Новый пользователь                       │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  git clone <repo>                                           │
│  pnpm install                                               │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  cp .env.development.local.example .env.development.local   │
│  (редактирует файл в своем редакторе)                      │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
         ┌──────────────────────────────────────┐
         │       pnpm setup                     │
         │  (node scripts/setup.mjs)            │
         └──────────────────────────────────────┘
         │
         ├─ Загружает loadEnv()
         │  из .env.development.local
         │
         ├─ Проверяет:
         │  ✓ DATABASE_URL?
         │  ✓ OPENAI_API_KEY?
         │  ✓ BETTER_AUTH_SECRET?
         │
         ├─ Вызывает init-db.mjs
         │  └─ loadEnv() из .env.development.local
         │  └─ Подключается к БД
         │  └─ Применяет миграции
         │
         └─ Показывает результат
                            │
                            ▼
        ✅ Проект готов к работе
           pnpm dev
```

## Компоненты системы

### 1. Загрузчик env-файлов (loadEnv)

```
Поиск переменной (например, DATABASE_URL):
├─ Ищет в .env.development.local     ← ПЕРВЫЙ приоритет
├─ Если не найдено, ищет в .env.local ← ВТОРОЙ приоритет
└─ Если не найдено, ищет в .env      ← ТРЕТИЙ приоритет
   
Результат: process.env.DATABASE_URL = "postgresql://..."
```

**Используется в:**
- `scripts/init-db.mjs` — инициализация БД
- `scripts/migrate.mjs` — применение миграций
- `scripts/setup.mjs` — комплексная проверка

### 2. Скрипт setup.mjs

```
┌─────────────────────────────┐
│   setup.mjs                 │
├─────────────────────────────┤
│ 1. loadEnv() — загружает    │
│    переменные из .env       │
│                             │
│ 2. Проверяет обязательные   │
│    переменные              │
│    - DATABASE_URL          │
│    - OPENAI_API_KEY        │
│    - BETTER_AUTH_SECRET    │
│                             │
│ 3. Выводит понятные ошибки  │
│    если что-то не хватает   │
│                             │
│ 4. Вызывает init-db.mjs     │
│    для инициализации БД     │
│                             │
│ 5. Показывает успех         │
└─────────────────────────────┘
```

### 3. Скрипт init-db.mjs

```
┌─────────────────────────────┐
│   init-db.mjs               │
├─────────────────────────────┤
│ 1. loadEnv()               │
│ 2. Проверяет DATABASE_URL  │
│ 3. Подключается к БД       │
│ 4. Читает SQL миграции      │
│    (drizzle/0000_*.sql)    │
│ 5. Применяет команды       │
│ 6. Обрабатывает ошибки     │
│    "таблица уже существует"│
│ 7. Показывает результат    │
└─────────────────────────────┘
```

### 4. Скрипт migrate.mjs

```
┌─────────────────────────────┐
│   migrate.mjs               │
├─────────────────────────────┤
│ 1. loadEnv()               │
│ 2. Проверяет DATABASE_URL  │
│ 3. Подключается к БД       │
│ 4. Использует drizzle-orm  │
│    migrator для применения │
│    всех миграций из        │
│    папки drizzle/          │
│ 5. Показывает результат    │
└─────────────────────────────┘
```

## Файлы конфигурации

```
.env.development.local    ← Личные настройки разработчика
                          ← Не коммитится в git
                          ← Имеет приоритет 1
                          
.env.local                ← Общие локальные настройки
                          ← Может коммититься в git
                          ← Имеет приоритет 2
                          
.env                      ← Общие настройки для всех
                          ← Коммитится в git
                          ← Имеет приоритет 3
                          
.env.example              ← Пример (без чувствительных данных)
.env.development.local.example ← Пример с подробными комментариями
```

## Ошибки и их обработка

### Ошибка: DATABASE_URL не установлена

```
Проверка (setup.mjs):
├─ Смотрит loadEnv() результат
├─ Видит что process.env.DATABASE_URL пустой
├─ Выводит сообщение об ошибке с указанием:
│  - Какая переменная не хватает
│  - Где её взять
│  - Какой файл отредактировать
└─ Завершает работу с кодом 1
```

### Ошибка: Не удается подключиться к БД

```
Проверка (init-db.mjs):
├─ DATABASE_URL есть
├─ Пытается подключиться
├─ Получает ошибку (например, ECONNREFUSED)
├─ Обнаруживает тип ошибки
├─ Выводит специфичный совет:
│  "⚠️ Не удается подключиться к БД. Проверьте DATABASE_URL"
└─ Завершает работу с кодом 1
```

### Предупреждение: Таблица уже существует

```
Применение миграции (init-db.mjs):
├─ Выполняет SQL команду CREATE TABLE
├─ Получает ошибку "already exists"
├─ Обнаруживает что это не критичная ошибка
├─ Логирует: "Таблица x уже существует, пропускаем"
├─ Продолжает со следующей команды
└─ Завершает успешно
```

## Преимущества архитектуры

| Аспект | Преимущество |
|--------|------------|
| **Автоматизация** | Не нужно вручную передавать DATABASE_URL |
| **Изоляция** | Каждый разработчик имеет личную конфигурацию |
| **Безопасность** | .env.development.local не коммитится в git |
| **Простота** | Одна команда вместо множества |
| **Информативность** | Подробные сообщения об ошибках |
| **Масштабируемость** | Легко добавить новые проверки |

## Интеграция с CI/CD

```
┌──────────────────────────────┐
│   GitHub Actions / GitLab CI │
├──────────────────────────────┤
│ 1. git clone                 │
│ 2. pnpm install              │
│ 3. pnpm db:migrate           │    ← Читает DATABASE_URL
│    (в CI есть env var)       │       из GitHub Secrets
│ 4. pnpm build                │
│ 5. pnpm start                │
└──────────────────────────────┘
```

В CI нет `.env.development.local`, поэтому:
- `loadEnv()` проверяет `.env.local`
- Затем проверяет `.env`
- Если не найдено, падает с понятной ошибкой

---

**Итого:** Архитектура достаточно гибкая для любого окружения:
- Local development (использует `.env.development.local`)
- Shared development (использует `.env.local`)
- Production (использует `.env` или переменные окружения)
