## 📖 Как это использовать

### Для новых пользователей, получивших проект

```bash
# 1. Клонируйте и установите зависимости
git clone <repo>
cd <project>
pnpm install

# 2. Создайте конфиг
cp .env.development.local.example .env.development.local
# Отредактируйте файл в любом редакторе и вставьте реальные значения

# 3. Инициализируйте проект
pnpm setup

# 4. Запустите
pnpm dev
```

### Для существующих разработчиков

Ничего не меняется! Все старые команды работают:
```bash
pnpm db:init      # Инициализирует БД (теперь читает .env файлы)
pnpm db:migrate   # Применяет миграции (теперь читает .env файлы)
pnpm db:push      # Работает как раньше
```

## 🔄 Порядок приоритета env-файлов

Когда Node.js скрипт ищет переменную окружения:

1. **`.env.development.local`** — используется, если существует (личные настройки разработчика)
2. **`.env.local`** — используется, если `.env.development.local` не существует
3. **`.env`** — используется, если остальные не существуют

**Первый найденный файл с переменной используется, остальные пропускаются.**

Это позволяет каждому разработчику иметь свой `.env.development.local` без конфликтов.

## ⚙️ Технические детали

### Функция loadEnv()

```javascript
function loadEnv() {
  const projectRoot = path.join(__dirname, '..');
  
  const envFiles = [
    path.join(projectRoot, '.env.development.local'),
    path.join(projectRoot, '.env.local'),
    path.join(projectRoot, '.env'),
  ];

  for (const envFile of envFiles) {
    if (fs.existsSync(envFile)) {
      const content = fs.readFileSync(envFile, 'utf-8');
      content.split('\n').forEach(line => {
        const [key, ...valueParts] = line.split('=');
        if (key && !key.startsWith('#') && key.trim()) {
          const value = valueParts.join('=').trim();
          const cleanValue = value.replace(/^["']|["']$/g, '');
          if (!process.env[key.trim()]) {
            process.env[key.trim()] = cleanValue;
          }
        }
      });
    }
  }
}
```

### Безопасность

- `.env.development.local` и `.env.local` **не коммитятся** в git (см. `.gitignore`)
- Каждый разработчик имеет свой личный env-файл
- Пароли и API ключи остаются только у владельца
- На production DATABASE_URL устанавливается в переменные Vercel
